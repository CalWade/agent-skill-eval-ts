# 实习产出说明

## 一、情景来源

### 业务背景

公司为客户提供自建定制的 AI Agent Skill 市场（基于 OpenClaw 平台）。客户的核心诉求有三个：

1. **Skill 质量保障** — SKILL.md 写出来之后，Agent 实际跑起来的效果好不好？
2. **模型选型** — 哪个顶尖模型效果好？kimi-k2、GPT-4o、DeepSeek 跑同一批 Skill，哪个成功率高、速度快？
3. **持续迭代** — 改了 SKILL.md 之后效果是好了还是坏了？能不能自动回归验证？

### 直接触发点

进行 Skill 编写时，发现客户使用较弱模型时，Skill 实现效果不稳定甚至完全不能用，而强模型对于客户而言成本过高，因此团队需要对已有 Skill 进行大规模的、各个模型的单独测试，以便为客户提供性价比平衡之选。**没有自动化手段持续检测 Skill 的质量，只能靠人肉试。**

---

## 二、项目定位

**这是一个 Agent 测试工具，不是 Agent 开发。**

做三件事：
1. 按 YAML 里写好的 prompt 逐条发给 Agent API
2. 拿到 Agent 的完整回复
3. 用规则检查回复是否符合预期，输出报告

附带：多模型横向对比、实时回显、汇总大屏。

类比前端的 E2E 测试（Playwright）：预设输入 → 执行操作 → 断言输出。被测对象从网页换成了 LLM Agent，断言方式从 DOM 查询换成了文本匹配。

## 三、设计思路

### 1. 双模式运行，适配云端和本地

**Cloud 模式**：对接任意 OpenAI 兼容 API，通过发送切换指令（如 `/model gpt-4o`）在同一端点内切换模型，一个 API Key 搞定所有模型。

**Local 模式**：对接本地 OpenClaw Gateway（`/v1/chat/completions`）。模型通过请求体 `model` 字段直接指定（格式 `providerId/modelId`），每次调用携带独立 `x-openclaw-session-key`（格式 `eval:{runId}:{modelId}:{caseId}`），三层隔离保证测试结果完全独立：
- `runId`：同一批次共享，不同批次不同
- `modelId`：不同模型互不影响
- `caseId`：同一模型不同用例之间无上下文污染

### 2. YAML 驱动，界面管理

测试用例用 YAML 声明式定义，不写代码。同时提供可视化界面增删改用例，保存时写回 YAML 文件，技术和非技术角色都能操作。

### 3. 简化判定策略

三种文本匹配判定类型（不引入 LLM-as-Judge）：

| 类型 | 说明 |
|------|------|
| `output_contains` | 回复必须包含指定文本 |
| `output_not_contains` | 回复不能包含指定文本 |
| `output_contains_any` | 回复包含列表中任意一个文本 |

无 `pass_criteria` 的用例标记为 `DISPLAY`，不做自动判定，只展示原文给人工判断。适用于开放式回复——LLM 的措辞不固定，人眼判断比文本匹配更可靠。

未来可引入 LLM-as-Judge 作为补充。

### 4. 副作用标记

用例通过 `side_effect` 字段声明：`none`（安全）/ `read`（只读）/ `write`（写入外部系统）。界面上标签着色区分，CLI 模式支持 `--safe-only` 只跑安全用例。

### 5. 多模型横向对比

一套用例自动依次跑多个模型，结果按模型 × 用例矩阵展示，直接对比通过率、耗时、token 消耗。Cloud 模式通过切换指令切换，Local 模式通过 model 字段 + sessionKey 保证每个模型结果完全独立。

### 6. 多步对话用例（steps）

用例支持 `steps` 字段定义多步连续对话。同一用例内所有步骤共享同一 `sessionKey`，Agent 保有上下文记忆，可测试「创建→编辑→查询」等跨轮依赖场景。

- steps 内每步有独立的 `instruction` 和 `pass_criteria`
- verdict 聚合：所有步骤全 PASS → PASS，任意 FAIL → FAIL
- 报告保留每步明细（`StepResult`），可定位到具体哪一步失败
- 向后兼容：现有单条 `instruction` 用例无需任何改动

```yaml
cases:
  - title: 创建并编辑飞书文档
    steps:
      - instruction: "新建一个飞书文档，标题是测试"
        pass_criteria:
          - type: output_contains
            text: "创建成功"
      - instruction: "在刚才的文档里追加 Hello World"
        pass_criteria:
          - type: output_contains
            text: "追加成功"
  # 单步用例完全兼容，无需改动
  - instruction: "查询我的飞书文档列表"
```

### 7. 从 CLI 到可视化大屏

项目迭代路径：

```
v0.1 Python 原型（CLI）
  → v0.2 TypeScript 重写（CLI，加判定引擎）
  → v0.3 Trace 解耦（确认平台不暴露 tool_calls）
  → v1.0 全栈 Web 版本（Vue 3 + Express）
  → v1.1 双模式运行（新增 Local/OpenClaw 模式，完全隔离上下文）
  → v1.2 多步对话（steps 编排 + 内联编辑 + 确认弹窗 + 深色主题统一）
```

| 前端 UI | Element Plus | 表单/对话框/表格/折叠组件齐全 |

| 包管理 | pnpm | 快，磁盘占用小 |

### 核心功能说明

**多步执行引擎（`services/runService.ts`）**

抽取 `executeCase()` 私有函数，单步/多步在此分叉，`executeRun` 主循环不感知 steps 细节：

```
executeRun(models, cases)
└─ 对每个模型：
   └─ 对每个 case：executeCase(case, sessionKey)
      ├─ 单步：callAgent(instruction) → judge() → CaseModelResult
      └─ 多步：for each step（共享 sessionKey）：
               callAgent(step.instruction) → judge(step.pass_criteria)
               → 聚合为一个 CaseModelResult
                 verdict: 任意 FAIL → FAIL，全 DISPLAY → DISPLAY
                 steps[]: 保留每步明细，带 [步骤N] 前缀的 failReasons
```

**用例内联编辑（`CasePanel.vue`）**

编辑不再弹窗，点「编辑」后卡片原地展开为编辑表单（琥珀色边框标识），确认/取消后折叠回展示态。支持单步和多步模式切换，多步模式下步骤可增删。

**全局确认弹窗（`ConfirmDialog.vue` + `useConfirm.ts`）**

替代 `ElMessageBox`，贴合深色工业风设计语言：深色背景、红色顶部警告条、毛玻璃遮罩。`useConfirm()` 返回 Promise，任意组件调用。当前用于：删除用例、清空日志。

## 六、明确的边界

**这个项目是什么：**
- Agent 结果断言器 / 测试工具
- 给 Agent 发固定的 prompt，检查回复是否符合预期
- 支持单步和多步连续对话两种用例形式
- 支持云端 API 和本地 OpenClaw 两种运行模式

**这个项目不是什么：**
- 不是 Agent 开发（没有自主决策、工具调用编排、多步执行）
- 不是性能压测工具（串行跑，不并发）
- 不是实时监控系统（跑完就结束，不常驻）
- 不做 Trace / 工具调用链分析（平台内部编排，不暴露 tool_calls）
