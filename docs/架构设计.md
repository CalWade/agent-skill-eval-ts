# 架构与功能设计全介绍

## 1. 整体架构

项目是一个 **全栈单体应用**，后端与前端通过 HTTP 通信，Vite 代理解决跨域问题，生产环境下可静态文件托管在同一 Express 进程中。

```
┌──────────────────────────────────────────────────────────────┐
│                         Browser                              │
│  Vue 3 (Vite + Element Plus + ECharts)   port :5173         │
│  ConfigPanel │ SuitePanel │ LiveLog │ ResultDashboard        │
└────────────────────────┬─────────────────────────────────────┘
                         │ HTTP / SSE
                         │ (Vite dev proxy → :3001)
┌────────────────────────▼─────────────────────────────────────┐
│                  Express API Server  port :3001              │
│  /api/config  /api/models  /api/suites  /api/suite          │
│  /api/run (SSE)  /api/reports  /api/report                  │
│                         │                                    │
│  ┌──────────────────────▼──────────────────────────────────┐ │
│  │                 Test Executor                           │ │
│  │  runner.ts → agent.ts → judge.ts                       │ │
│  └──────────────────────────────────────────────────────── ┘ │
└───────────────────────────────────────────┬──────────────────┘
                                            │ HTTP POST
                                            ▼
                                  Agent API (OpenAI 兼容)
                                  所有模型共用同一端点
```

### 技术选型

| 层级 | 选型 | 说明 |
|------|------|------|
| 后端运行时 | Node.js + tsx | tsx 直接运行 TypeScript，无需编译 |
| 后端框架 | Express | 轻量，SSE 实现简单直接 |
| 前端框架 | Vue 3 + Vite | Composition API + setup script |
| 前端 UI | Element Plus | 表单/对话框/表格组件齐全 |
| 前端图表 | ECharts | 通过率柱状图 + 耗时柱状图 |
| 类型共享 | 手动镜像 | `server/types.ts` ↔ `web/src/types/index.ts` |
| 样式 | CSS（scoped）+ 全局覆盖 | 深色主题，#0f1117 背景 |
| 持久化 | 文件系统 | `.env` / `models.json` / YAML / JSON 报告 |
| 实时推流 | SSE（Server-Sent Events）| 比 WebSocket 轻，单向推送足够 |

## 2. 后端模块详解

### 2.1 `server/api.ts` — Express 服务器

整个后端的唯一入口文件，承担 HTTP 请求处理和测试执行编排。

**配置管理（`.env`）**：
- `readEnv()` 手动解析 `.env` 文件，不依赖 `process.env`（避免启动时加载顺序问题）
- `writeEnv()` 直接写回文件，同时更新 `process.env` 保证当次运行生效

**模型管理（`models.json`）**：

两层 fallback 机制：
```
models.json（用户自定义）→ server/models.ts（硬编码默认）
```

```typescript
interface ModelsFile { prefix: string; ids: string[] }
// 文件格式示例：{ "prefix": "/model", "ids": ["gpt-4o", "kimi-k2"] }

getModels()      // 返回 ModelConfig[]，供测试执行使用
getModelsFile()  // 返回原始 { prefix, ids }，供前端编辑回显
```

`ModelConfig.switchCmd` 在读取时动态拼接：`${prefix} ${id}`，不存储完整指令，只存前缀和 id，避免冗余。

**测试执行（`POST /api/run`）**：

SSE 头部设置后立即 `flushHeaders()`，之后通过 `send(event, data)` 写入事件流。执行流程：

```
对每个模型：
  1. emit model_start
  2. 发送 switchCmd → callAgent() → emit switch_ok / switch_fail
  3. 切换失败 → 该模型所有用例标记 FAIL，continue
  4. 对每个用例：
       callAgent(instruction) → judge() → emit case_result
  5. emit model_done
完成：构建 EvalReport → 写 JSON 文件 → emit done
```

所有等待（`sleep()`）在 `intervalMs` / `switchWaitMs` > 0 时插入，最后一条/最后一个模型不等待。

### 2.2 `server/agent.ts` — Agent API 调用

单次无状态 HTTP 请求封装：

- 请求体格式：标准 OpenAI `{ messages, stream: false, ...extraBody }`
- 超时控制：`AbortController` + `setTimeout`，触发则返回错误的 `CallResult`
- 响应解析：兼容 `choices[0].message.content`（标准 OpenAI）和直接返回 `content`（部分平台变体）
- Token 统计：从 `usage.prompt_tokens`、`usage.completion_tokens` 提取，若平台不返回则为 `null`
- 全部字段包装进 `CallResult`，调用方拿到的是确定性结构，不会 throw

**每次调用都是全新会话**（`messages` 只含当前一条），平台侧靠 `instance_id` 维持模型状态（切换后保持）。

### 2.3 `server/judge.ts` — 判定引擎

简单文本匹配，三种类型：

| `type` | 逻辑 | `verdict` 影响 |
|--------|------|----------------|
| `output_contains` | `output.includes(text)` | 不包含 → `failReasons` |
| `output_not_contains` | `!output.includes(text)` | 包含 → `failReasons` |
| `output_contains_any` | `texts.some(t => output.includes(t))` | 全不含 → `failReasons` |

`pass_criteria` 为空或 undefined → `verdict = 'DISPLAY'`（只展示，不判断）
`failReasons.length === 0` → `verdict = 'PASS'`
`failReasons.length > 0` → `verdict = 'FAIL'`

### 2.4 `server/config.ts` — 配置加载

从 `process.env` 读取，`AGENT_API_URL` 和 `AGENT_API_KEY` 缺失时抛错。

`AGENT_EXTRA_BODY` 支持任意 JSON 字符串，合并进每次请求的 body，用于传递平台特有参数（如 `instance_id`）。同时单独处理 `AGENT_INSTANCE_ID`，方便最常见的 `instance_id` 场景配置。

### 2.5 `server/models.ts` — 默认模型列表

硬编码兜底，当 `models.json` 不存在或解析失败时使用。当前列表：

```
gpt-4o / gpt-4o-mini / deepseek-v3 / deepseek-r1 / kimi-k2 / claude-3-5-sonnet
```

同时提供 `filterModels(ids)` 和 `listModelIds()` 工具函数供 CLI 使用。

### 2.6 `server/runner.ts` — CLI 执行器（备用）

仅 CLI 模式使用，与 `api.ts` 中的执行逻辑平行但独立。输出到 stdout，使用 `report.ts` 中的格式化函数打印进度。

### 2.7 `server/cli.ts` — 命令行入口（备用）

通过命令行参数控制测试范围（`--suite`、`--models`、`--safe-only`、`--dry-run`）。在有可视化界面后，CLI 模式作为无头批量运行的备选方式保留。

## 3. 前端模块详解

### 3.1 整体布局（`App.vue`）

```
┌─ header（顶栏）──────────────────────────────────────────┐
│  平台标题  │  Tab: 运行测试 / 历史报告                    │
├─ left-panel（380px）────┬─ right-panel（flex: 1）────────┤
│  ConfigPanel            │  LiveLog                       │
│  SuitePanel             │  ResultDashboard               │
│                         │                                │
│  （历史 Tab 时）         │  （历史 Tab 时）               │
│  HistoryPanel           │  ResultDashboard（历史数据）    │
└─────────────────────────┴────────────────────────────────┘
```

`App.vue` 负责：
- Tab 切换控制
- SSE 事件消费：调用 `startRun()`，将事件流转换为结构化 `LogEntry[]`
- 测试完成后接收 `EvalReport` 传给 `ResultDashboard`
- 历史报告通过 `HistoryPanel` emit 传入同一个 `ResultDashboard`

### 3.2 `ConfigPanel.vue` — API 配置

表单字段：API URL、API Key（密码框）、Instance ID、请求间隔、切换等待、超时秒数。

- `onMounted` → `GET /api/config` 填充表单
- 保存 → `POST /api/config` 写入 `.env`，同时 Express 侧更新 `process.env`，无需重启

### 3.3 `SuitePanel.vue` — 测试配置核心

最复杂的组件，包含三个子区域：

**模型区**：
- 从 `GET /api/models` 加载，展示为复选框，默认勾选前两个
- 「编辑列表」弹出 Dialog：显示当前前缀和 id 列表，支持增删，右侧实时预览拼接结果
- 保存 → `POST /api/models` → 重新加载列表 → 移除已删除模型的勾选状态

**套件区**：
- 下拉选择已有 YAML 文件，切换时自动加载 `GET /api/suite?file=xxx`
- 「新建」按钮清空当前表单
- 保存套件 → `POST /api/suite`，文件名自动使用 `skill` 名称

**用例区**：
- 列表展示 id、title、instruction、副作用标签、判定条件数量
- 「编辑」弹出用例 Dialog：id、title、instruction、side_effect 下拉、pass_criteria 列表
  - 切换 criteria 类型时自动重置字段（`output_contains_any` 用逗号分隔字符串 ↔ `texts[]`）
- 「删除」直接 `splice`
- 「添加用例」自动生成 `TC-XX` 格式 id

**开始测试按钮**：
- `canRun` = `selectedModelIds.length > 0 && suite.cases.length > 0 && suite.skill.length > 0`
- emit `run` 事件给 App.vue 触发 SSE 请求

### 3.4 `LiveLog.vue` — 实时日志

- 接收 `LogEntry[]`，按 `type` 着色：`info`（灰）/ `success`（绿）/ `error`（红）/ `muted`（暗）
- 监听 `logs.length` 变化，`nextTick` 后自动滚动到底部
- 等宽字体，仿控制台视觉效果

### 3.5 `ResultDashboard.vue` — 汇总大屏

接收 `EvalReport`，渲染四个区域：

**统计卡片行**：
- 每个模型一张卡，显示 `passed/judged`、通过率、平均耗时（秒）、总 token 数
- 通过率 100% 绿色，否则橙色

**ECharts 图表行**：
- 通过率柱状图：纵轴 0-100%，每模型一条
- 平均耗时柱状图：单位秒
- 每次 `report` 变化时重新 `init`（先 `dispose` 旧实例）

**用例对比表**：
- 行 = 用例，列 = 模型
- 单元格组件 `CellResult.vue`：显示 verdict 图标（✅/❌/📋）+ 耗时 + token 数
- `row-fail` 样式标红整行（任意模型 FAIL）

**回复详情折叠列表**：
- `el-collapse`，默认折叠
- 每个用例展开后显示 `reply-grid`：每个模型一列，使用 `ReplyBlock.vue`

### 3.6 `CellResult.vue` — 对比表单元格

compact 显示：verdict 图标 + 耗时（秒）+ token 数（有则显示）。

### 3.7 `ReplyBlock.vue` — 回复详情块

完整展示单条结果：
- verdict 标签（颜色对应状态）
- 调用失败时显示错误信息
- 成功时显示原始回复文本（`pre-wrap`，限高 200px 可滚动）
- `failReasons` 列表（橙色警告）

### 3.8 `HistoryPanel.vue` — 历史报告列表

- `onMounted` → `GET /api/reports` 获取报告文件名列表
- 点击文件名 → `GET /api/report?file=xxx` → emit `view(report)` 给 App.vue
- 当前选中项高亮

### 3.9 `api/index.ts` — 前端 API 封装

所有请求通过 Vite dev proxy 转发到 `:3001`，路径前缀 `/api`。

SSE 处理（`startRun()`）：
1. `fetch POST /api/run` 拿到 `response.body`（ReadableStream）
2. `getReader()` 逐块读取，`TextDecoder` 解码
3. 按 `\n\n` 切分 SSE 事件块，提取 `event:` 和 `data:` 行
4. `JSON.parse(data)`，合并 `type: event` 后调用 `onEvent()`
5. 收到 `done` 调用 `onDone()`，返回 `cancel()` 函数供外部中断

## 4. 数据流

### 4.1 启动测试完整流程

```
用户点击「开始测试」
  → SuitePanel emit('run', { modelIds, suite })
  → App.vue handleRun()
  → api.startRun() → POST /api/run
  → Express：
      ① 验证参数
      ② 读取 .env → loadConfig()
      ③ 读取 models.json → getModels()
      ④ 过滤 selectedModels
      ⑤ 对每个模型：
           发 switchCmd → callAgent() → SSE switch_ok/fail
           对每个用例：
             callAgent(instruction) → judge() → SSE case_result
      ⑥ 构建 EvalReport → 写 JSON → SSE done
  → 前端 SSE 处理：
      每个事件 → addLog() 追加日志
      收到 done → report.value = e.report → ResultDashboard 展示
```

### 4.2 模型编辑流程

```
用户点击「编辑列表」
  → GET /api/models → { models, prefix, ids }
  → 填充 editingPrefix / editingIds
  → 用户修改后点「保存」
  → POST /api/models { prefix, ids }
  → 写 models.json
  → 重新 GET /api/models 刷新 models.value
  → 已删除模型从 selectedModelIds 移除
```

### 4.3 持久化文件一览

| 文件 | 格式 | 写入时机 | 读取时机 |
|------|------|---------|---------|
| `.env` | KEY=VALUE | 保存 API 配置时 | 每次 `POST /api/run` 前 |
| `models.json` | JSON `{ prefix, ids }` | 保存模型列表时 | 每次 `GET /api/models`、`POST /api/run` |
| `examples/**/*.yaml` | YAML | 保存测试套件时 | 选择套件时、启动测试时 |
| `results/*.json` | JSON（`EvalReport`）| 每次测试完成后 | 历史报告页面点击时 |

## 5. 核心类型

```typescript
// 判定类型（不含 LLM-as-Judge，只做文本匹配）
type Verdict = 'PASS' | 'FAIL' | 'DISPLAY'

// 单次调用结果（agent.ts 返回）
interface CallResult {
  success: boolean;
  output: string;          // 回复原文
  durationMs: number;
  promptTokens: number | null;
  completionTokens: number | null;
  totalTokens: number | null;
  finishReason: string | null;
  error: string | null;
}

// 一个用例 × 一个模型 的结果
interface CaseModelResult {
  caseId: string; caseTitle: string; modelId: string;
  call: CallResult;
  verdict: Verdict;
  failReasons: string[];   // FAIL 时列出未通过的条件描述
}

// 完整测试报告（存 JSON 的就是这个）
interface EvalReport {
  skill: string; description: string; timestamp: string;
  modelIds: string[];
  cases: Array<{ id; title; instruction; sideEffect }>;
  results: CaseModelResult[];   // M×N 展平列表
}
```

## 6. 设计决策说明

**为何不用 LLM-as-Judge？**
测试 API 配额有限，每次调用都已消耗资源。引入 judge 模型会把请求量翻倍，且引入额外不确定性。当前方案：有判定条件用文本匹配，无判定条件直接展示原文让人工判断，足够用于快速冒烟测试。

**为何每次调用都是新会话（无历史 messages）？**
模型切换通过向 Agent 发送指令实现，平台侧维持 session 状态。每个测试用例的 `instruction` 是独立的意图，不依赖上下文。单轮调用更稳定，也便于独立统计每条用例的 token 消耗。

**为何用 SSE 而非 WebSocket？**
测试执行是单向推送（服务器 → 浏览器），SSE 是 HTTP 原生支持的单向流，实现更简单，不需要维护双向连接状态。浏览器兼容性与 WebSocket 相当。

**为何不用 Pinia/Vuex？**
数据流清晰：`App.vue` 是唯一的状态协调点，子组件通过 props/emit 通信，不需要全局 store。引入状态管理库会增加复杂度但带来的收益不明显。

**models.json 为何只存 prefix + ids，不存完整 switchCmd？**
减少冗余。完整指令 = prefix + 空格 + id，是派生值，读取时动态拼接，避免前缀改变时需要逐一更新每条记录。
