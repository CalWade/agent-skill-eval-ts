# 架构与功能设计全介绍

## 1. 整体架构

项目是一个**全栈单体应用**，后端与前端通过 HTTP 通信，Vite 代理解决跨域问题。

```
┌──────────────────────────────────────────────────────────────┐
│                         Browser                              │
│  Vue 3 (Vite + Element Plus + ECharts)   port :5173         │
│  ConfigPanel │ SuitePanel │ CasePanel │ LiveLog │ Dashboard  │
└────────────────────────┬─────────────────────────────────────┘
                         │ HTTP / SSE
                         │ (Vite dev proxy → :3001)
┌────────────────────────▼─────────────────────────────────────┐
│                  Express API Server  port :3001              │
│  /api/config  /api/models  /api/localModels                 │
│  /api/suites  /api/suite  /api/run (SSE)                    │
│  /api/reports  /api/report                                  │
│                         │                                    │
│  ┌──────────────────────▼──────────────────────────────────┐ │
│  │             services/runService.ts                      │ │
│  │   executeCase() → agent.ts → judge.ts                  │ │
│  │   openclawService.ts（Local 模式隔离）                  │ │
│  └─────────────────────────────────────────────────────────┘ │
└──────────────────┬────────────────────────┬──────────────────┘
                   │ Cloud 模式             │ Local 模式
                   ▼                        ▼
         Agent API（OpenAI 兼容）   OpenClaw Gateway :18789
         切换模型：发 switchCmd     切换模型：models set + secrets reload
         上下文：平台管理           隔离：workspace 备份还原 + sessionKey
```

### 技术选型

| 层级 | 选型 | 说明 |
|------|------|------|
| 后端运行时 | Node.js + tsx | 直接运行 TypeScript，无需编译步骤 |
| 后端框架 | Express | 轻量，SSE 实现简单直接 |
| 前端框架 | Vue 3 + Vite | Composition API + setup script |
| 前端 UI | Element Plus | 表单/对话框/表格组件齐全 |
| 前端图表 | ECharts | 通过率柱状图 + 耗时柱状图 |
| 类型共享 | 手动镜像 | `server/types.ts` ↔ `web/src/types/index.ts` |
| 样式 | CSS scoped + 全局覆盖 | 深色工业主题，`industrial.css` 统一覆盖 Element Plus |
| 持久化 | 文件系统 + localStorage | `.env` / `models.json` / YAML / JSON 报告 / 勾选状态 |
| 实时推流 | SSE（Server-Sent Events）| 单向推送，比 WebSocket 实现简单 |

## 2. 后端模块详解

### 2.1 `server/api.ts` — Express 服务器

路由注册与请求/响应处理，业务逻辑委托给 `services/` 层。

**POST /api/run（SSE 测试执行）**：

设置 `text/event-stream` 头后立即 `flushHeaders()`，通过 `send(event, data)` 写入事件流：

```
对每个模型：
  1. emit model_start
  2. Cloud：发 switchCmd → callAgent() → emit switch_ok / switch_fail
     Local：openclawService.switchModel() → emit switch_ok / switch_fail
  3. 切换失败 → 该模型所有用例标记 FAIL，continue
  4. 对每个用例：executeCase() → emit case_result
  5. emit model_done
完成：构建 EvalReport → 写 JSON 文件 → emit done
```

**GET /api/localModels**：

读取 `~/.openclaw/openclaw.json` 获取所有 provider 配置，并发调用每个 provider 的 `/v1/models` 接口获取完整模型列表。若 provider 不支持该接口，回退到 `openclaw.json` 中的静态配置。返回格式：`[{ id: "providerId/modelId", name, provider }]`。

**配置读写**（`POST /api/config`）：

采用**合并写入**策略：前端只发送当前模式可见的字段，服务端读取现有 `.env` 后与新字段合并，再整体写回，避免切换模式时另一模式的配置被清空。

### 2.2 `server/services/configService.ts` — 配置服务

```typescript
readEnvFile()   // 读 .env → Record<string, string>
writeEnvFile()  // 合并写入 .env + 同步 process.env
loadConfig()    // 从 process.env 构建 AppConfig，按 mode 分支校验
```

`loadConfig()` 按模式分支：

- **cloud 模式**：校验 `AGENT_API_URL` 和 `AGENT_API_KEY` 必填
- **local 模式**：校验 `LOCAL_TOKEN` 必填，`LOCAL_BASE_URL` 默认 `http://127.0.0.1:18789`

### 2.3 `server/services/modelService.ts` — 模型服务

两层 fallback：`models.json`（用户自定义）→ `server/models.ts`（硬编码）。

`models.json` 格式：

```json
{
  "prefix": "/model",
  "ids": [
    "custom-dashscope/qwen3.5-plus",
    "qingshu/gpt-5.3"
  ]
}
```

Local 模式下，`ids` 中的值本身就是 `provider/modelId` 格式，直接用于 OpenClaw 模型切换，不需要额外的映射字段。

### 2.4 `server/services/openclawService.ts` — OpenClaw 管理

负责 Local 模式下的模型切换和 workspace 隔离，是 Local 模式隔离机制的核心。

**模型切换**：

```typescript
switchModel(providerModelId: string): Promise<void>
```

执行 `openclaw models set <providerModelId>` + `openclaw secrets reload`，实现 gateway 热重载，无需重启进程。整个切换约 2 秒完成。

**Workspace 隔离**：

```typescript
backupWorkspace(runId: string): string      // 备份，返回备份目录路径
resetFromBackup(backupDir: string): void    // 从备份还原（不删除备份，可反复调用）
restoreWorkspace(backupDir: string): void   // 还原并清理备份目录
```

备份范围：
- `~/.openclaw/workspace/`（IDENTITY.md、SOUL.md、USER.md 等）
- `~/.openclaw/agents/main/sessions/`（所有对话历史 .jsonl 文件）

备份存放在系统临时目录 `{tmpdir}/openclaw-eval-backup/{runId}/`。

### 2.5 `server/services/runService.ts` — 执行引擎

SSE 路由和 CLI 共用的核心逻辑，通过回调接口解耦输出方式。

**核心设计：`executeCase()` 私有函数**

单步/多步在此分叉，`executeRun` 主循环不感知 steps 细节：

```typescript
// 单步路径
const callResult = await callAgent(c.instruction, agentConfig)
const { verdict, failReasons } = judge(callResult.output, c.pass_criteria)
return { ..., call: callResult, verdict, failReasons }

// 多步路径（steps 共享同一 sessionKey）
for (const step of c.steps) {
  const callResult = await callAgent(step.instruction, agentConfig)
  const { verdict, failReasons } = judge(callResult.output, step.pass_criteria)
  stepResults.push({ stepIndex, instruction, call: callResult, verdict, failReasons })
}
// 聚合：任意 FAIL → FAIL；全 DISPLAY → DISPLAY；否则 PASS
// failReasons 带 [步骤N] 前缀；call 取最后一步（向后兼容）
```

**Local 模式完整执行流程**：

```
1. getDefaultModel()              记录当前默认模型
2. backupWorkspace(runId)         备份 workspace + sessions 作为基准快照

对每个模型：
  3. resetFromBackup(backupDir)   还原到基准快照（所有模型从同一状态出发）
  4. switchModel(model.id)        切换 gateway 默认模型（models set + secrets reload）
  对每个用例：
    5. executeCase(case, sessionKey)  单步或多步执行
    6. emit case_result

finally：
  7. switchModel(originalModel)   恢复原始默认模型
  8. restoreWorkspace(backupDir)  还原 workspace 到测试前状态，清理备份
```

**隔离保证**：
- 模型间：每个模型从相同的 workspace 快照出发，前一个模型的任何写入对下一个不可见
- 用例间（单步）：独立 `sessionKey`，对话历史不跨用例共享
- 步骤间（多步）：同一用例内 steps 共享 `sessionKey`，形成连续对话
- 批次间：`runId` 基于时间戳生成，不同测试批次 sessionKey 完全不同

### 2.6 `server/agent.ts` — Agent 调用

支持 cloud / local 双模式，统一入口 `callAgent()` 内部自动分派：

**Cloud 模式（`callAgentCloud`）**：

- 标准 OpenAI 格式：`{ messages, stream: false, ...extraBody }`
- 每次只发当前 user 消息（无状态单轮）
- `AbortController` 控制超时

**Local 模式（`callAgentLocal`）**：

- 端点：`{localBaseUrl}/v1/chat/completions`
- 请求体：`{ model: "openclaw", messages, stream: false }`
- 额外 header：
  - `Authorization: Bearer {localToken}`
  - `x-openclaw-session-key: {sessionKey}`

注：`model` 字段固定为 `"openclaw"`，底层实际使用哪个 LLM 由 gateway 当前默认模型决定（由 `openclawService.switchModel` 提前切好）。

### 2.7 `server/judge.ts` — 判定引擎

三种文本匹配，无 LLM-as-Judge：

| `type` | 逻辑 | 判定影响 |
|--------|------|---------|
| `output_contains` | `output.includes(text)` | 不包含 → failReasons |
| `output_not_contains` | `!output.includes(text)` | 包含 → failReasons |
| `output_contains_any` | `texts.some(t => output.includes(t))` | 全不含 → failReasons |

`pass_criteria` 为空 → `DISPLAY`；`failReasons` 为空 → `PASS`；否则 `FAIL`。

## 3. 前端模块详解

### 3.1 整体布局

```
┌─ header（顶栏）─────────────────────────────────────────┐
│  [AGENT SKILL EVAL]  │  STATUS  │  时钟  │  HISTORY 按钮 │

│  ConfigPanel      │  CasePanel        │  LiveLog         │

└───────────────────┴───────────────────┴──────────────────┘
```

History 通过右侧抽屉（`el-drawer`）展示。报告大屏通过路由 `/report` 独立页面展示。

### 3.2 `ConfigPanel.vue`

模式切换 radio group（Cloud / Local），根据 `agentMode` 条件渲染不同字段：

- Cloud：API URL、API Key、Instance ID、切换等待
- Local：Gateway URL、Gateway Token

**保存时始终发送完整表单**（包含两种模式的字段），后端合并写入，避免切换模式后另一模式配置丢失。

### 3.3 `SuitePanel.vue`

**模型编辑弹窗**根据当前 `agentMode` 显示不同内容：

- **Cloud 模式**：切换指令前缀 + 模型名列表（纯文本输入）
- **Local 模式**：供应商下拉（从 `/api/localModels` 拉取）+ 模型 ID 输入框，保存时自动组合为 `providerId/modelId`

**勾选状态持久化**：`selectedModelIds` 写入 `localStorage`，key 为 `selectedModelIds`，页面刷新后自动恢复，仅保留当前模型列表中仍存在的 id。

### 3.4 `CasePanel.vue` — 内联编辑

用例列表支持**卡片内联编辑**，不弹窗：

- 点「编辑」：卡片原地展开为编辑表单，琥珀色边框标识编辑态
- 点「确认」/「取消」：折叠回展示态
- 编辑表单支持**多步模式开关**：
  - 关闭（默认）：单条 instruction + pass_criteria
  - 开启：steps 列表，每步独立 instruction 和 pass_criteria，可增删步骤

删除用例前弹出确认弹窗（带用例名称），防止误操作。

### 3.5 `ConfirmDialog.vue` + `useConfirm.ts` — 全局确认弹窗

替代 Element Plus `ElMessageBox`，贴合深色工业风：

- 深色背景（`#1c2128`）+ 红色顶部警告条
- monospace 大写标题 + 毛玻璃遮罩
- `useConfirm()` 返回 Promise，`await confirm({...})` 即可调用

```typescript
// 任意组件使用
const confirm = useConfirm()
await confirm({ title: '删除用例', message: '确认删除「xxx」？', confirmText: '删除' })
```

当前使用处：删除用例、清空日志。

### 3.6 `ResultDashboard.vue` — 汇总大屏

回复详情区支持多步展示：多步用例按步骤分块展开，每步显示独立 verdict 标签（PASS/FAIL/DISPLAY）、指令内容和 Agent 回复。

### 3.7 `web/src/styles/industrial.css` — 深色主题

全局覆盖 Element Plus 默认白色样式，统一为深色工业风：

## 4. 数据流

### 4.1 启动测试完整流程（Local 模式）

```
用户点击「开始测试」
  → POST /api/run { modelIds, suite }
  → Express：
      ① loadConfig()：mode=local，读取 localBaseUrl / localToken
      ② getModels()：获取 provider/modelId 格式的模型列表
      ③ runService.executeRun()：
           getDefaultModel()       记录原始模型
           backupWorkspace()       备份 workspace + sessions

           对每个模型：
             resetFromBackup()     还原到初始快照
             switchModel(id)       models set + secrets reload
             对每个用例：
               executeCase(case, sessionKey)
               ├─ 单步：callAgent → judge → CaseModelResult
               └─ 多步：逐步 callAgent（共享 sessionKey）→ 聚合 CaseModelResult
               → SSE case_result

           switchModel(original)   恢复原始模型
           restoreWorkspace()      还原并清理备份
      ④ buildReport() → 写 JSON → SSE done
  → 前端：SSE 事件 → addLog() + ResultDashboard
```

### 4.2 配置保存流程

```
用户修改配置 → 点保存
  → POST /api/config（完整表单，含两种模式字段）
  → writeEnvFile()：readEnvFile() merge 新数据 → 写 .env + 更新 process.env
  → 下次 loadConfig() 立即生效，无需重启
```

### 4.3 持久化文件一览

| 文件 | 格式 | 写入时机 | 读取时机 |
|------|------|---------|---------|
| `.env` | KEY=VALUE | 保存 API 配置时（合并写入） | 每次 POST /api/run 前 |
| `models.json` | JSON `{ prefix, ids }` | 保存模型列表时 | GET /api/models、POST /api/run |
| `suites/**/*.yaml` | YAML | 保存测试套件时 | 选择套件时、启动测试时 |
| `results/*.json` | JSON（EvalReport）| 每次测试完成后 | 历史报告页面点击时 |
| `logs/server.log` | 纯文本 | 服务端每条 console.log | 开发调试 |
| `localStorage.selectedModelIds` | JSON 数组 | 勾选状态变化时 | 页面加载时 |

## 5. 核心类型

```typescript
type AgentMode = 'cloud' | 'local'

interface TestStep {
  instruction: string;
  pass_criteria?: PassCriteria[];
}

interface TestCase {
  id?: string;
  title?: string;
  instruction?: string;   // 单步必填；steps 存在时可省略
  steps?: TestStep[];     // 多步用例；步骤间共享 sessionKey
  side_effect?: SideEffect;
  pass_criteria?: PassCriteria[];  // 单步判定条件；steps 存在时忽略
}

/** 多步用例中单步的执行结果 */
interface StepResult {
  stepIndex: number;
  instruction: string;
  call: CallResult;
  verdict: Verdict;
  failReasons: string[];
}

interface CaseModelResult {
  caseId: string; caseTitle: string; modelId: string;
  call: CallResult;       // 单步结果；多步时取最后一步（向后兼容）
  verdict: Verdict;
  failReasons: string[];  // 多步时带 [步骤N] 前缀
  steps?: StepResult[];   // 多步用例时填充
}

interface EvalReport {
  skill: string; description: string; timestamp: string;
  modelIds: string[];
  cases: Array<{
    id: string; title: string; instruction: string;
    sideEffect: SideEffect;
    stepCount?: number;   // 多步用例步骤数
  }>;
  results: CaseModelResult[];
}
```

## 6. 设计决策

**Local 模式为何用 `models set + secrets reload` 而非其他方式切换模型？**

经过实测，OpenClaw gateway 忽略请求体中的 `model` 字段和 `x-openclaw-agent-id` header——前者不起作用，后者只切换 workspace 不切换底层 LLM。唯一能真正切换底层 LLM 的方式是修改全局默认模型配置。`secrets reload` 命令可以让 gateway 热重载配置，无需重启进程，约 2 秒完成。

**为何需要备份还原 workspace，而不只是隔离 sessionKey？**

`sessionKey` 只隔离对话历史（sessions），但 workspace 中的 `IDENTITY.md`、`SOUL.md`、`USER.md` 等文件会被 agent 读写，模型 A 的测试写入的内容会影响模型 B 的行为。备份还原确保每个模型从完全相同的初始状态出发。

**多步 steps 为何共享 sessionKey 而不是独立 sessionKey？**

steps 的核心价值是测试跨轮依赖——Agent 能否记住上一步的结果并连贯操作。独立 sessionKey 会切断上下文，退化成多条独立单步用例，失去多步测试的意义。

**为何用内联编辑而非弹窗？**

弹窗遮挡其他用例，编辑时无法对比其他用例内容。内联展开让编辑和列表并存，空间利用更合理，操作上下文更清晰。

**为何用自定义 ConfirmDialog 而非 ElMessageBox？**

`ElMessageBox` 样式硬编码为白色，无法通过 CSS 变量覆盖至深色主题。自定义组件完全掌控样式，与整体工业风设计语言一致。

**writeEnvFile 为何改为合并写入？**

前端 ConfigPanel 根据当前模式只渲染/发送对应字段，若覆盖写入会清空另一模式的配置。合并写入保留历史配置，两种模式可以自由切换而不丢失各自的字段。

**为何用 SSE 而非 WebSocket？**

测试执行是单向推送（服务器 → 浏览器），SSE 是 HTTP 原生支持的单向流，实现更简单，不需要维护双向连接状态。

**为何不用全局状态管理（Pinia）？**

数据流清晰：`MainPage.vue` 是唯一状态协调点，子组件通过 props/emit 通信，不需要全局 store。
